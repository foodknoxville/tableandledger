They let calculators into the ACT. Nobody mourned long division.

The same thing is happening to basic programming right now, and the institutions have not caught up yet. Universities are still selling four-year computer science degrees as the golden ticket while the industry they are supposedly preparing students for is actively automating the entry-level work those degrees qualify you to do.

## What Changed

Two years ago, if you wanted to build a simple automation -- pull data from an API, process it, send a notification -- you needed to know a programming language well enough to write, debug, and deploy the script. That was a real barrier. It kept a lot of useful work locked behind a skill that took months or years to develop.

That barrier is gone. Not lowered. Gone.

An operator with a clear understanding of what they need can describe a workflow to an AI assistant and get working code back in minutes. Not pseudocode. Not a tutorial. Working, deployable code that does the thing. The food cost tracking system I wrote about in my last post was built this way. I did not write the Zapier integrations or the API calls by hand. I described what I needed, validated the output, and deployed it.

The skill that mattered was not writing code. The skill that mattered was knowing what to ask for and being able to evaluate whether the answer was right.

## The Skill That Actually Matters

The argument for learning to code was never really about the code itself. It was about computational thinking -- understanding logic, sequences, conditions, loops, data structures. Knowing how systems work. Knowing what is possible and what is not.

That is still valuable. That will always be valuable. But the implementation layer between "I understand what needs to happen" and "a computer does the thing" is collapsing rapidly. The value is moving up the stack, from writing the instructions to understanding the problem well enough to describe it.

For a restaurant operator, this is actually great news. You do not need to learn Python to build an invoice processing pipeline. You need to understand your invoice processing workflow well enough to specify it clearly. You already have that knowledge. You have had it for years. The missing piece was never your coding skills. It was affordable, accessible tools that could translate your operational knowledge into automation.

## What Institutions Should Be Teaching Instead

If I were advising a university right now, I would not cut computer science programs. I would restructure them. The fundamentals still matter -- algorithms, data structures, systems thinking, computational logic. But the emphasis on writing code in specific languages as the primary skill is increasingly misplaced.

What operators and business professionals actually need:

Systems thinking. How to decompose a complex process into discrete, automatable steps. How to identify where the bottlenecks are. How to think about data flow.

Specification literacy. The ability to describe what you want a system to do with enough precision that an AI or a developer can build it correctly. This is harder than it sounds and almost nobody teaches it.

Evaluation skills. When an AI gives you code, can you tell if it is right? Can you test it against edge cases? Can you identify where it might fail? This is the new debugging -- not finding syntax errors, but validating logic against domain knowledge.

Integration thinking. Understanding how different tools and APIs connect, what the constraints are, what the failure modes look like. Not how to write the integration code, but how to architect the integration.

## The Point

The most dangerous advice you can give someone entering the workforce right now is "learn to code" as if that is a durable competitive advantage. The advantage is knowing your domain well enough to leverage the tools that are eliminating the need to code.

That is what I tell restaurant operators who ask me about AI. You do not need to become a programmer. You need to become a better operator -- one who can articulate what you need with enough clarity that the tools can build it for you. The operators who thrive in the next five years will not be the ones who learned Python. They will be the ones who understood their own operations deeply enough to specify what they needed.

The code is the easy part now. The knowledge never was.
